        API Reference DeepLynx v0.0.1

        Modules

        Datum

        Datum keeps the contexts that define your domain and business logic.

        Datum.Accounts

        The Accounts context.

        Datum.Accounts.Group

        Groups are user or system defined groupings of users. Groups do not inherently have any permissions in the system - they are only used to group users together when setting permissions in other systems. end

        Datum.Accounts.User
        Datum.Accounts.UserGroup

        User Groups are just the collection of membership for a particular group. At time of creation, there are currently no plans to have roles inside the groups. end

        Datum.Accounts.UserNotifier
        Datum.Accounts.UserToken
        Datum.Agent

        Agent is a GenServer (https://hexdocs.pm/elixir/GenServer.html) which is in charge of running the AI Assistant for Datum. Each instance of the GenServer is an individual agent, interacting with one single individual user.

        Datum.Common

        The Common context. Dealing with the models that don't fit nicely into other contexts, such as the explorer tab context.

        Datum.Common.ExplorerTabs

        ExplorerTabs are the representation of an open viewing tab in the data explorer home. This system allows us to easily save and restore a user's workspaces without having to use javascript or local storage garbage. Sqlite is more than fast enough, especially with LiveView to handle restore/close options.

        Datum.DataOrigin

        The DataOrigin context. See data_origin/Origin.ex for information on what a DataOrigin is and how it slots into the bigger picture of Datum.

        Datum.DataOrigin.CT

        Needed for the closure_table library. NOTE: You MUST call this only after you've done put_dynamic_repo, either yourself or through the function above. If not, you're potentially writing a previous connection's database. I know, it's potentially dangerous - but so is breathing.

        Datum.DataOrigin.Data

        Data is the actual stored data for an origin, typically represents a file or directory, but more types are supported and will be supported as time continues.

        Datum.DataOrigin.DataHistory

        This structure is populated automatically in the database when data is inserted. Keeping the schema separate allows us to know when we're working with historical data.

        Datum.DataOrigin.DataSearch

        Reflects the virtual table for FTS5 searching.

        Datum.DataOrigin.DataTreePath

        DataTreePath is for use with the closure_tables library. It's needed to hold the relationship information for us modeling a filesystem

        Datum.DataOrigin.Documentation

        Documenation for a piece of data. It's stored in a separate table vs. embedded schema so that we can do things like semantic search on it since there will often be more than one piece of documentation.

        Datum.DataOrigin.ExtractedMetadata

        Playing around with this currently - we're thinking that we need the extracted metadata to be decomposed or normalized into the database structure so that we can do semantic search either via FTS5 or vector search. Definitely a work in progress structure.

        Datum.DataOrigin.Origin

        Represents a point of origin for data. Typically we view these as file systems, network drives, HPC clusters etc. It's a logical, hierarchical separation of data that is representive of a filesystem. Origins should contain a lot of data - but if they start to grow too large we'll need to look at separating them or figuring how to handle splitting the underlying Sqlite3 database. That's right - each Origin has it's OWN Sqlite3 databased stored at the configured filesystem point.

        Datum.DataOrigin.Origin.AzureConfig

        Azure Blob Configuration for Data Origins

        Datum.DataOrigin.Origin.DuckDBConfig

        DuckDB Configuration for Data Origins

        Datum.DataOrigin.Origin.FilesystemConfig

        Filesystem configuration - not necessarily needed if the filesystem is local, but if it's a networked location - we need the information to make connections. Note: that we can't actually easily make that connection from within Erlang itself.

        Datum.DataOrigin.Origin.R2Config

        R2 Configuration for Data Origins

        Datum.DataOrigin.Origin.S3Config

        S3 Configuration for Data Origins

        Datum.DataOrigin.OriginRepo

        This is the repo for interacting with DataOrigin's Sqlite3 databases. Each Data Origin will have it's own, separate Sqlite3 database and so we will use dynamic repositories to work with it.

        Datum.Duckdb

        DuckDB is a GenServer (https://hexdocs.pm/elixir/GenServer.html) which is in charge of running any queries on CSV and Parquet files currently. Each instance of this GenServer is an individual DB, interacting with a single user through a LiveView.

        Datum.Mailer
        Datum.ModuleName

        This is how we store and restore Elixir module names in our application. Typically used for our explorer window panes or plugin names. This type is used by Ecto to safely store and restore Strings to Atoms.

        Datum.Permissions

        The Permissions context. See the permissions folder for information.

        Datum.Permissions.Data

        Permissions table for Data in a DataOrigin- reference this table whenever you're loading data. Try to do so as part of the query itself, not a separate call.

        Datum.Permissions.DataOrigin

        Permissions table for DataOrigins - reference this table whenever you're loading origins. Try to do so as part of the query itself, not a separate call.

        Datum.Plugins

        The Plugins context. See the plugins/Plugin.ex file for information on what a plugin is. These are all the functions for interacting with the Plugin model in the database

        Datum.Plugins.CSV

        General metadata extraction for Parquet files

        Datum.Plugins.Extractor

        Extractor module for running either the plugin system for extracting metadata or native elixir code to do the same.

        Datum.Plugins.Parquet

        General metadata extraction for Parquet files

        Datum.Plugins.Plugin

        Plugin represents a plugin supplied either by the system or by the user. These plugins are typically used for file system scanning and metadata extraction. Plugins can be WASM modules, either raw binary or on disk, or Elixir modules implementing the correct plugin type behavior.

        Datum.Plugins.TdmsIndex

        General metadata extraction for TDMS index files

        Datum.Release

        Used for executing DB release tasks when run in production without Mix installed.

        Datum.Repo

        This is meant to be the "operations" repo. This should hold user information, information on permissions, plugins, metadata models, maybe some pointers to data platforms - BUT NOT THE DATA ITSELF.

        Datum.Scanner

        This is a GenServer in charge of running scan processes from the CLI.
        If you are wanting the server or other processes to run scanning - prefer using the scanner you need directly.

        Datum.Scanners.Filesystem

        Filesystem scanner is the primary scanning tool used by the CLI when on a local system. Feed it the root path and it will recursively run through the contained folders/files and run all enabled plugins on them.

        Datum.Search

        Search is a GenServer (https://hexdocs.pm/elixir/GenServer.html) which is in charge of dispatching a user search across all the data origins - because each origin is it's own database we had a few ways of approaching this - from attaching and combining tables in a single sqlite3 db, to doing this concurrent search. We chose the concurrent search because there might be hundreds or more data origins and that could easily break an SQL query, not to mention potentially swamp a single database.

        DatumWeb

        The entrypoint for defining your web interface, such as controllers, components, channels, and so on.

        DatumWeb.AgentLive

        This is the tab for interacting with an AI agent, if the integration is enabled and a suitable configuration (like Ollama) is present. Currently we do only support OpenAI enabled endpoints (which Ollama supports).

        DatumWeb.CLIChannel
        DatumWeb.CLISocket
        DatumWeb.Client

        This is the HTTP client for the application.

        DatumWeb.CoreComponents

        Provides core UI components.

        DatumWeb.Endpoint
        DatumWeb.ErrorHTML

        This module is invoked by your endpoint in case of errors on HTML requests.

        DatumWeb.ErrorJSON

        This module is invoked by your endpoint in case of errors on JSON requests.

        DatumWeb.Gettext

        A module providing Internationalization with a gettext-based API.

        DatumWeb.GraphExplorerLive

        This is the live component for exploring data in a graph like manner. Typically started with either one or a group of a data. This component allows users to traverse their data in a graph and make connections between pieces of data manually (planned, not currently implemented)

        DatumWeb.HomeLive

        This is the main navigation area after a user logs in. Consists of multiple panes of tabs, each tab should be its own separate live_view. Try to do as little as possible on this page, though it will probably get more complicated as we add inter-view communication.

        DatumWeb.Layouts

        This module holds different layouts used by your application.

        DatumWeb.LiveComponent.ConnectData

        This is used to connect two pieces of data, even between separate origins. This is used to create connections in the relationships field of %Data{} not connecting in the directory/file paradigm

        DatumWeb.LiveComponent.DeleteOrigin

        This is used to delete a data origin

        DatumWeb.LiveComponent.TDMSMetadata

        This is used to display TDMS metadata, if present, within the File Viewer on the Data Origin Explorer.

        DatumWeb.OriginController

        The json endpoints for working with Data Origins

        DatumWeb.OriginCreationLive

        This is the tab for adding data.

        DatumWeb.OriginExplorerLive

        This is our primary way of exploring a Data Origin. It is very similar to a file explorer, as it needs to represent directories and the files in those directories.

        DatumWeb.PageController
        DatumWeb.PageHTML

        This module contains pages rendered by PageController.

        DatumWeb.PluginsController

        The json endpoints for handling pluging information and binary module retrieval. Typically used by the CLI tool to sync for local use.

        DatumWeb.Router
        DatumWeb.SearchLive

        This is our tab for running searches across the entirety of the Datum ecosystem. If state exists, only hold the state of the query itself and filters.

        DatumWeb.SocketClient

        This is a SlipStream websocket client designed for the CLI to interact with the central server.

        DatumWeb.Telemetry
        DatumWeb.UserAuth

        This module contains mostly user authentication middleware needed by various routes

        DatumWeb.UserConfirmationInstructionsLive
        DatumWeb.UserConfirmationLive
        DatumWeb.UserForgotPasswordLive
        DatumWeb.UserLoginLive
        DatumWeb.UserRegistrationLive
        DatumWeb.UserResetPasswordLive
        DatumWeb.UserSessionController
        DatumWeb.UserSettingsLive
        DatumWeb.WikiLive

        This is the live view for rendering the internal Wiki. The wiki lives in /priv/wiki. The wiki requires a home.md and table_of_contents.md to function correctly. Pages are rendered by treating the url after wiki/ as a param representing the wiki pages file name. So in order to work, you must have a wiki page who's file name is URL formatted.

        Mix Tasks

        mix database.clean

        Cleans the local database files.

        mix wiki.gen

        Generates a new wiki page and adds to the table of contents. Name should be the full, captilized name of the post - e.g Table of Contents
